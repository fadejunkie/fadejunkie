# Lobe Growth Log

This file tracks Lobe's accumulated design knowledge, pattern discoveries, and aesthetic learnings across every session. Lobe updates this file after every task.

## Design Patterns Discovered
- **`var(--font-geist-mono)` is the ghost token — always use `var(--font-mono)`:** In FadeJunkie, the canonical monospace token is `var(--font-mono)`. Any occurrence of `var(--font-geist-mono)` is wrong — it references a Next.js internal variable name that may not be exposed as a CSS custom property in all contexts. Detection: grep `font-geist-mono` across all component files before any polish pass. Found in `PublicStatusBadges.tsx` — a pure display component built in isolation without referencing sibling components. The false negative: it works in some environments if `--font-geist-mono` happens to be set, masking the bug.
- **`<h2>`/`<h3>` section label violations span all component types — audit status ecosystem too:** The [HARDENED] rule "section labels must use `<p>` not heading tags" was confirmed again across 4 components in the status ecosystem: `StatusPanel`, `StatusHistory`, `ConnectionsInbox` (all using h2 for section/group headers). The global CSS applies `text-transform: lowercase; letter-spacing: -0.04em` to all h2/h3 — while `lowercase` may be intentional for editorial titles, the `letter-spacing: -0.04em` on small text and the semantic incorrectness are problems. The fix is always `<p>` with explicit `fontFamily` and `textTransform: "none"` when needed for proper nouns.
- **`min-h-[44px]` on flex rows containing toggles/actions = reliable mobile touch target fix:** When a flex row contains a small visual toggle (20px) alongside label text, adding `min-h-[44px]` to the row and `items-center` (not `items-start`) expands the tappable area for the entire row without changing the visual size of the toggle. This is cleaner than the `-m + p` padding expansion approach (which fights background color rendering) and cleaner than pseudo-element hit targets. Apply to any card row where the primary action is a small toggle or button.
- **Skeleton loading in DiscoveryFeed replaces plain "Loading..." text:** When the main content (`statuses`) is loading (paths already resolved), show 4 skeleton cards in the same grid layout as the real cards — same `rounded-xl border border-border bg-card p-4` container, `animate-pulse` divs for each content zone. For the initial paths load, also skeleton the filter tab strip (4 pills of varying widths). The result is zero layout shift when content arrives and visual consistency with MatchesFeed. Pattern: extract a `SkeletonCard` component at the top of the file, reuse it for both loading states.
- **Card headers missing font-heading are always in Courier Prime — it's not obvious:** `<p className="text-lg font-bold text-foreground">Your Profile</p>` renders in Courier Prime because `font-bold` is a weight utility, not a font-family utility. The global CSS doesn't cascade font-family to `<p>` tags — only h1–h4. So ANY `<p>` used as a section header without explicit `style={{ fontFamily: "var(--font-heading)..." }}` will render in body font. This is the "implicit League Spartan" trap — devs assume it inherits from the parent or the heading rules, but `<p>` is explicitly excluded from the global heading CSS. Detection: grep `<p className="text-lg font-` or `<p className="text-base font-` in page/card files and check for font-heading style.


- **Pure display component for cross-context reuse — no `"use client"` needed:** When a component just renders data from props (no hooks, no interactivity), omitting `"use client"` lets it be used in both server components (barber profile, shop page) and client components (PostCard) without restriction. The client-side data-fetching concern is lifted into a thin wrapper (`PostStatusIndicator`) that adds `"use client"` + `useQuery`. Pattern: `PublicStatusBadges` (pure display, no directive) + `PostStatusIndicator` ("use client", calls useQuery, renders PublicStatusBadges). The display component handles both compact/full modes via prop.
- **`Promise.all` for parallel server-side fetches in server components:** When a Next.js server page needs data from two independent Convex queries, `const [a, b] = await Promise.all([fetchQuery(...), fetchQuery(...)])` fires them in parallel instead of serial. In the barber profile, gallery + status summary now resolve together — halving the waterfall. Apply this pattern any time you add a second `fetchQuery` to an existing server page.
- **Convex `useQuery` deduplication on authorId across feed posts:** Calling `useQuery(api.statuses.getPublicStatusSummary, { userId: authorId })` per post in the feed is safe because Convex deduplicates subscriptions — multiple components querying the same key share one subscription. The `PostStatusIndicator` per-post pattern is correct; no manual caching or batching needed. Render it at the component level, let the Convex client handle deduplication transparently.
- **`color-mix(in oklch, var(--foreground) 18%, transparent)` for subtle borders:** In a B&W system, `border: "1px solid color-mix(in oklch, var(--foreground) 18%, transparent)"` is the token-correct replacement for hardcoded `rgba(0,0,0,0.18)`. It inverts properly in dark mode (18% of white instead of 18% of black), keeps the visual subtlety, and uses the CSS token system. Use this pattern for any subtle border that should adapt to theme.
- **Always-render Sheet vs conditional render — the animation trap:** When a Sheet (or any Radix Dialog) is conditionally rendered with `{open && <Sheet>}`, closing the sheet causes the component to unmount immediately — Radix never gets to run its `data-[state=closed]` exit animation. The fix: always render the Sheet component in the tree and pass `open` as a controlled prop. Radix's internal state machine handles mount/unmount timing. The mutation hook inside is trivially cheap to always mount; don't sacrifice animation correctness to avoid it.
- **`{sheetOpen && <ConnectSheet>}` will silently kill slide-out animations:** The parent's conditional removes the component before Radix can fire `data-[state=closed]:slide-out-to-bottom`. The component must stay in the tree. This is the most common mistake when building "lazy-mounted" modal patterns in React — lazy mount by avoiding expensive work inside, not by removing the Dialog from the DOM.
- **Ghost connect button: `/45` opacity + no border = secondary without visual weight:** A connect button at `text-muted-foreground/45 hover:text-foreground` with no border and a 12px icon is pure secondary — it doesn't compete with the card's primary content hierarchy. The hover transition to full foreground gives clear affordance without a button frame. This pattern is the "link-style action" in a card context — use it for any non-primary card action.
- **`hasInitialized` ref guard for batch auto-mark-seen on mount:** When an inbox component should mark all pending items as seen on first view, use `const hasMarkedRef = useRef(false)` with `if (!connections || hasMarkedRef.current) return`. Set `hasMarkedRef.current = true` before firing the mutations. This ensures the batch runs exactly once even if the query re-fires on focus/reconnect. Pattern: `pending.forEach(c => mutation({ id: c._id as Id<"table"> }).catch(() => {}))` — fire-and-forget with swallowed errors since visibility marks are non-critical.
- **ConvexError `.data` property for client-side error messages:** When a Convex mutation throws `new ConvexError("message")`, the client catches it as `{ data: string }`. The safe pattern: `(err as { data?: string })?.data ?? (err instanceof Error ? err.message : "Something went wrong")`. This chain handles both ConvexError (data property) and generic Error (message property). Match on message substrings: `.includes("Already sent")`, `.toLowerCase().includes("no longer active")`.
- **`<blockquote>` as note/quote block style:** For a user's note message in an inbox row, `<blockquote className="border-l-2 border-border pl-3 text-sm text-muted-foreground">` is the most semantic and visually restrained way to show authored content. The 2px border-l in border color is subtle enough to read as "quoted" without being heavy. Font-body (Courier Prime) inside the blockquote signals authentic authored content vs. system metadata.
- **Left accent bar as pending state signal in B&W systems:** A `w-0.5 self-stretch rounded-full bg-foreground` div as the leftmost element of an inbox row creates a clean "unread/pending" signal without color. When seen: `bg-transparent`. The transition between these two states (via className swap) is instant — which is fine since it auto-marks on mount. The bar takes zero visual space when transparent, so read items collapse cleanly into the divide-y rhythm.
- **Unseen badge on section headers — inverted pill:** `bg-foreground text-background text-[9px] font-semibold` in a `min-w-4 h-4 px-1 rounded-full` pill next to the section heading is the FJ pattern for unread counts. The inversion (dark on light in light mode) is stronger than any color badge in a B&W system. Cap at "9+" to keep the pill size stable.
- **Tab-on-page pattern — server component keeps metadata, client component owns tabs:** When adding tab-based navigation to an existing Next.js App Router server page, keep the page file as a server component (so `metadata` export still works) and extract all client interactivity into a new `<XTabs>` client component. The page renders the tabs component as a child. `DiscoverTabs` is the canonical FJ pattern: `useState<"browse" | "matches">` + tab bar + conditional render. Zero logic duplication — DiscoveryFeed and MatchesFeed are separate client trees.
- **Tab bar: `-mb-px` border-b trick for seamless active underline:** Tab button with `pb-2.5 -mb-px border-b-2 border-foreground` on active / `border-transparent` on inactive. The container gets `border-b border-border`. The active tab's `border-b-2` sits at the same bottom position as the container border, visually overriding it (the active tab becomes part of the frame). `transition-colors duration-150` makes the switch feel instant but not harsh.
- **Two empty states for contextual discovery views:** When a query can be empty for two distinct reasons (1. user has no prerequisites activated vs 2. prerequisites exist but no complementary matches), run a secondary lightweight query (`getMyStatuses`) to distinguish them. Show different copy + different affordance for each: "activate statuses + link to /status" vs "check back later." Never collapse them to one generic empty state — the first requires action, the second is passive.
- **`showMatchIndicator` prop pattern — progressive component enhancement:** Rather than creating a separate `MatchCard` component that duplicates `DiscoveryCard`'s internals, add an optional `showMatchIndicator?: boolean` prop to the existing card. When true, renders a contextual icon (Link2) in the bottom row. This keeps a single card component as the source of truth, zero duplication, and the indicator is opt-in at the usage site.
- **`humanizeKeyLower()` for section headers vs `humanizeKey()` for card labels:** When the same key appears as a section header ("because you're **seeking apprenticeship**"), use `.replace(/_/g, " ").toLowerCase()` — no title-casing, reads as natural sentence continuation. When it appears as a standalone label ("Seeking Apprenticeship"), use the Title Case helper. Two different visual registers for the same data source.
- **Read-only history rows — `divide-y divide-border/40` as the rhythm system:** Compact read-only lists (status history, audit logs, etc.) work best without card wrappers per row. A single `divide-y divide-border/40` on the container + `py-2.5` on each row creates clean row separation without visual weight. The 40% opacity divider is subtle enough to not compete with the text hierarchy, but enough to give the eye a rest between items.

Combined with `mb-2.5` below it, the group structure is clear without needing cards or borders per group.
- **Duration badge as `border border-border rounded px-1.5 py-0.5`:** A subtle bordered pill for duration metadata ("3d", "12h") in `text-[10px]` mono with `text-muted-foreground/50` creates a system-native "badge" feel without any background fill. Lower opacity than the timestamp alongside it creates a natural left-to-right weight hierarchy: label (most weight) → timestamp (medium) → duration (least).
- **`@/components/` vs `app/components/` — the alias trap:** In FJ, the `@/` alias maps to the project root (`C:\Users\twani\fadejunkie\app`), so `@/components/X` resolves to `C:\Users\twani\fadejunkie\app\components\X`. Creating a file at `app/app/components/` is a wrong-nest error — always verify by checking where existing sibling components live (e.g., `Glob('**/StatusPanel.tsx')` confirms the correct directory before writing the new file).
- **Discovery feed: `useQuery(..., "skip")` until path is selected:** When a Convex query depends on a piece of state that initializes async (e.g., `selectedPath` defaults to `null` until `getDiscoveryPaths` loads), pass `"skip"` as the second argument to `useQuery` instead of an invalid args object. `useQuery(api.statuses.discoverStatuses, selectedPath ? { path: selectedPath } : "skip")` — this prevents a query with undefined/null args from firing and producing an error. Once `selectedPath` is set, the real query fires reactively.

The `selectedPath === null` guard prevents the effect from resetting user selection when the query re-fires on focus/reconnect. `null` as initial state (not `""` or `undefined`) makes the guard unambiguous.

- **Initials avatar fallback — `bg-muted` + `text-muted-foreground` in mono:** For users without avatars, a circle with their initials in Geist Mono at `text-[11px] font-semibold` reads as intentional and on-brand. - **Prop-variant `isDark` vs system dark mode — the hidden double failure:** A component with `isDark = variant === "dark"` that uses hardcoded `"#ffffff"`/`"#000000"` for text colors has two separate failure modes: (1) It bypasses the CSS token system (the [HARDENED] baseline violation). (2) It conflates the card's visual variant with system dark mode — in system dark mode, a `light`-variant card has `isDark = false`, so text renders as `"#000000"` (black) on the now-dark page background = invisible text. The fix is always `isDark ? "var(--background)" : "var(--foreground)"` — these tokens resolve correctly for both system modes AND the card's variant intent. Observed: `components/ui/bento-grid.tsx` lines 80 and 107 (cycle 2026-03-28T19:45).
- **Tailwind arbitrary shadow values are hardcoded-color violations:** `hover:shadow-[0_2px_8px_rgba(0,0,0,0.06)]` in a Tailwind className is in the same violation category as `hover:border-[#c8c8c8]` — both bypass the CSS token system and hardcode a light-mode-only value. The FJ globals.css registers `--shadow-xs` through `--shadow-2xl` in `@theme inline`, which makes `shadow-sm`, `shadow-md`, etc. available as proper token-backed Tailwind classes. Use those. Never use `shadow-[rgba()]` arbitrary syntax. Detection: grep for `shadow-\[` in className strings.

Progress bars: always `bg-foreground` fill — the width is the data, color adds nothing. Flagged state: `border-foreground border-2` — thicker border = visual weight = "marked." - **Semantic color badges — B&W system enforcement:** In a zero-hue design system, `bg-amber-100 text-amber-800` badges are the most common sneaky violation — they look "soft" so they feel harmless, but they break the grayscale contract. Replacement map: amber "Featured" → `variant="secondary"` (neutral gray pill), "Verified" → `variant="outline"` (subtle border pill). - **Raw `border border-foreground` in inline buttons vs system Button `border-2 border-foreground`:** Even when a developer copies the button style manually and gets the color right, they'll often miss the `border` → `border-2` upgrade. The 1px border is half the weight of the system spec. Always grep for `border border-foreground` (not `border-2`) when hunting for visual inconsistencies — it's the most common "almost right" pattern.
- **`#000000` is the most dangerous hardcoded color — it's invisible in light mode:** `#000000` looks identical to `var(--foreground)` in light mode, so it survives code review. But in dark mode, `--foreground` becomes white while `#000000` stays black — instant invisible text. The tab switcher in `signin/page.tsx` had this exact bug: active tab text and its underline border were both `#000000`. Replaced with `var(--foreground)` (active) and `var(--muted-foreground)` (inactive). Lesson: when scanning for hardcoded colors, `#000000` and `#ffffff` are higher priority than any other hex value because their mode-mismatch failures are catastrophic (invisible), not just aesthetically wrong.
- **Nav link `opacity: 0.75` default is a trap:** It reads as "subtle polish" but it's actually a contrast failure on most displays. The hover state going to full opacity creates a disconcerting jump. Full opacity by default = no jump, links are immediately readable. Subtle polish belongs in micro-interactions (hover lift, underline reveals), not in baseline readability.
- **`border` → `border-2` on outline buttons — the single highest-ROI contrast fix:** `border border-foreground` (1px) on a white background is invisible at a glance — it looks faint and gets lost in card edges. `border-2 border-foreground` (2px) makes the button unmistakably interactive. This is the #1 button contrast fix in any B&W design system. Always specify `border-2` as the default for outline variants, never `border` alone.

## Techniques to Revisit
_Added when a technique showed promise but wasn't fully explored._
- Canvas halftone gives way more control than CSS radial-gradient hacks — can do per-dot sizing, staggered grids, and directional density gradients that CSS masks can't express cleanly
- Browser chrome mockups as UI artifacts: the key realism lever is the traffic light dots — they need to be the exact macOS colors AND reveal their glyphs (×, −, ↗) only on parent hover, not always visible. That single interaction pattern is what makes it read as "real window" vs "card with colored circles."
- **Web Animations API for sequenced UI choreography:** `.animate()` with `fill: 'forwards'` + `.onfinish` chaining gives precise control over multi-phase sequences (dismiss → pause → reappear). Always `.cancel()` the animations at the end to release property ownership back to CSS. Guard with a boolean flag to prevent re-entry during the sequence.
- **Design editor realism:** What makes selection chrome look "real" vs decorative is the layered specificity — two overlapping selection boxes at slightly different insets (parent frame vs child element), each with their own color system, plus the micro-details: resize cursors on handles, transform origin crosshairs, a frame label above the artboard, and a monospace dimension readout below. The subtle pulse on the border sells "actively selected" without being distracting.
- **Resize handle pattern:** Use `data-dir` attributes on handles + event delegation on the parent selection box. One `mousedown` listener reads the direction, then a single `mousemove` handler branches on it. Corner handles compute new width from dx, derive height from aspect ratio. Edge handles only mutate their axis. Always track both start position AND start dimensions — repositioning the opposite edge is what makes tl/bl/tr resize feel natural (you offset left/top by the delta between old and new dimensions).
- **CSS custom property as a resize bridge:** When a widget's content area height is set via CSS (transition-driven expand/collapse), use a CSS custom property (`--bw-content-h`) as the bridge between JS resize logic and CSS layout. JS sets the property on the widget element during resize; CSS `var()` consumes it in both the `height` transition rule and the inner page height. This keeps the expand/collapse transition working normally while allowing resize to override the default.
- **CSS 3D flip — instant reset on navigation:** When navigating between flashcards, if currently flipped, set `transition: 'none'` on the card inner, remove the flipped class, force a reflow with `void el.offsetHeight`, then restore transition to `''`. This prevents the user from seeing a backwards flip animation when moving between cards — the reset is invisible and the new card appears front-facing.
- **CSS class-toggle transitions for content rotation:** Instead of Web Animations API, simple CSS transitions on opacity + transform driven by adding/removing classes (`ds-exit`, `ds-enter`) give clean two-phase swap animations. The trick: add exit class → wait for transition → swap content + add enter class → force reflow → remove enter class (letting the default non-transformed state be the transition target). Lighter than WAAPI for simple fade-slide swaps.
- **Progress bar reset trick:** Set `transition: none` + `width: 0`, force reflow with `void el.offsetWidth`, then set `transition: width Xms linear` + `width: 100%`. The reflow between no-transition and transition is what makes the reset invisible and the fill smooth.
- **CSS transitions kill live resize dragging.** If a child element has `transition: height 400ms` and you're updating its height target (via CSS custom property) on every mousemove, the 400ms transition means the element is perpetually lagging behind the mouse — making it look like the resize is broken. Fix: set `element.style.transition = 'none'` on mousedown, restore to `''` on mouseup. This pattern applies to any drag interaction that fights a CSS transition.
- **Neon glow recipe:** Layered `text-shadow` is the entire trick — 3-4 stops from tight (6-8px) to wide (50-80px), each progressively lower opacity. The color should match the text `color` exactly for realism. `box-shadow` with the same approach works for glowing borders/lines.
- **Neon flicker animation:** Irregular opacity dips at the tail end of a long cycle (4-5s) with very short dip durations (1-2% of cycle each). Multiple dips with varying depths (0.65-0.88) reads as electrical instability without being annoying. A separate gentler "hum" animation (0.97 opacity at 50%) on secondary elements adds ambient life.
- **Sign-switching blackout:** When rotating content in a neon sign, a 100-120ms full-opacity-0 gap between exit and enter animations sells the "sign powering off then back on" metaphor. Without it, the crossfade reads as a normal card transition.
- **macOS folder shape in pure CSS:** `::before` pseudo-element with `border-radius: 8px 8px 0 0` + `border-bottom: none` positioned at `top: -18px` creates a seamless folder tab. The 1px height overlap between tab and body hides the seam. Subtle blue-tinted border on both tab and body creates a cohesive folder "sheen" without needing actual blue fill. The tab being slightly lighter than the body (`#12121e` vs `#0f0f1a`) adds just enough depth to read as a separate plane.
- **Dynamic DOM refs for rebuildable UIs:** When a widget's innerHTML gets cleared and rebuilt (e.g., win state → play again), `const` element refs go stale. Use getter functions (`function $el() { return document.getElementById('id'); }`) instead — they always return the live element. Negligible perf cost for widgets with infrequent lookups.
- **CDN React + Babel standalone for no-build dashboards:** `<script type="text/babel">` with Babel standalone compiles JSX in-browser — zero build step. Style objects over CSS-in-JS libs keeps the single-file constraint clean. CSS custom properties for the palette + inline style objects for component styles = best of both worlds. For developer tools, density > breathing room — tight padding (6-8px rows), 11px font, monospace everywhere, and warm near-black backgrounds keep it functional without looking cold/clinical.

Base palette: near-void (#080503) → glass layers at 4%/7%/11% warm-white opacity → border at 8%/14%. The `inset box-shadow: 0 1px 0 rgba(255,244,234,0.06)` top highlight is what makes glass cards read as raised — without it they look like flat rectangles. `backdrop-filter: blur(20px)` on cards, `blur(24px)` on sticky nav.
