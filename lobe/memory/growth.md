## Design Patterns Discovered
- **`<Card className="rounded-*" style={{ borderRadius }}>` is a high-frequency dead-className trap on collapsible card roots:** The Radix UI `Card` component does NOT strip `style` props ‚Äî they propagate verbatim to the root div. So `<Card className="rounded-xl shadow-sm" style={{ borderRadius: '0.75rem' }}>` has two conflicting rules for the same property: `rounded-xl` (0.75rem) from className and `borderRadius: '0.75rem'` from inline style. The inline style wins the CSS cascade every time, making the Tailwind class invisible dead code. Developers write this because they add `rounded-*` to className for readable intent AND pass `borderRadius` as a style prop as a "safety net" ‚Äî not realizing the Tailwind class is already doing the job (or rather, trying to). The fix is always to remove the inline `borderRadius` entirely ‚Äî the `rounded-*` class handles it. Detection: grep for `<Card className=.*rounded.*style=.*borderRadius` across all TSX files. Also applies to any other Radix UI primitive (`Sheet`, `Dialog`, `Popover`) that passes style props to a root div. Found in `components/PathSelector.tsx:244`. (cycle 2026-03-29T20:00)

- **`style={{ width, height }}` on Lucide icons is a structural inline-style violation ‚Äî use `w-* h-*` Tailwind classes instead:** Lucide React icons accept either a `size` prop or `className` for sizing. Using `style={{ width: N, height: N }}` directly on an icon element is the same category as `style={{ borderRadius }}` or `style={{ letterSpacing }}` ‚Äî a structural CSS property with a direct Tailwind class equivalent, delivered via inline style instead. The violation is especially easy to miss because icon sizing isn't typography-related and doesn't show up in the fontFamily audit. The fix: move width/height from style prop into className using `w-* h-*` utilities. Size mapping: 11px ‚Üí `w-[11px] h-[11px]`, 12px ‚Üí `w-3 h-3`, 14px ‚Üí `w-3.5 h-3.5`, 16px ‚Üí `w-4 h-4`, 20px ‚Üí `w-5 h-5`. When the icon already has a className (for color/opacity), simply prepend the size classes and delete the style prop. Detection: grep `style={{ width` in all TSX ‚Äî every icon hit should be converted. Found in `components/DiscoveryCard.tsx`: three icons (`Link2`, `Check`, `UserPlus`) each with their own `style={{ width, height }}` ‚Äî all fixed in one commit. (cycle 2026-03-29T19:45)

- **Dark-section total-saturation: `#000000` bg + `rgba(255,255,255,*)` text is a compound fingerprint for fully-hardcoded template components:** When a third-party or template-derived component sets `backgroundColor: "#000000"` on a section AND uses `rgba(255,255,255,N%)` for every single text and border color tier, the entire component is hardcoded to a dark-only mental model and will never render correctly in a multi-theme token system. In light mode the dark footer visually works but is semantically wrong; in dark mode the `#000000` bg blends into the page and the `rgba(255,255,255,*)` text on a black page background has no contrast to offer. Detection: grep the file for `#000000` + `rgba(255,255,255` ‚Äî if both patterns coexist in the same component, assume total saturation and plan a comprehensive pass. Fix protocol: `backgroundColor: "#000000"` ‚Üí `bg-foreground` Tailwind class (inverts correctly: black in light mode / white in dark mode); ALL `rgba(255,255,255,N%)` ‚Üí `color-mix(in oklch, var(--background) N%, transparent)` (since `--background` = white in light, the color-mix output is white-tinted = correct; in dark mode `--background` = black so the text inverts to black-tinted, preserving readability on the now-white footer surface); `onMouseEnter`/`onMouseLeave` JS style mutations also need updating since they bypass React's render cycle and hardcode the rgba values. `components/ui/shadcnblocks-com-footer2.tsx` was the canonical case: 8 rgba(255,255,255,*) values across logo text, tagline, column headers, nav links, hover states, bottom border, copyright, and legal links ‚Äî all fixed in one pass. ## Techniques to Revisit
_Added when a technique showed promise but wasn't fully explored._
- Canvas halftone gives way more control than CSS radial-gradient hacks ‚Äî can do per-dot sizing, staggered grids, and directional density gradients that CSS masks can't express cleanly
- Browser chrome mockups as UI artifacts: the key realism lever is the traffic light dots ‚Äî they need to be the exact macOS colors AND reveal their glyphs (√ó, ‚àí, ‚Üó) only on parent hover, not always visible. That single interaction pattern is what makes it read as "real window" vs "card with colored circles."
- **Web Animations API for sequenced UI choreography:** `.animate()` with `fill: 'forwards'` + `.onfinish` chaining gives precise control over multi-phase sequences (dismiss ‚Üí pause ‚Üí reappear). Always `.cancel()` the animations at the end to release property ownership back to CSS. Guard with a boolean flag to prevent re-entry during the sequence.
- **Design editor realism:** What makes selection chrome look "real" vs decorative is the layered specificity ‚Äî two overlapping selection boxes at slightly different insets (parent frame vs child element), each with their own color system, plus the micro-details: resize cursors on handles, transform origin crosshairs, a frame label above the artboard, and a monospace dimension readout below. The subtle pulse on the border sells "actively selected" without being distracting.
- **Resize handle pattern:** Use `data-dir` attributes on handles + event delegation on the parent selection box. One `mousedown` listener reads the direction, then a single `mousemove` handler branches on it. Corner handles compute new width from dx, derive height from aspect ratio. Edge handles only mutate their axis. Always track both start position AND start dimensions ‚Äî repositioning the opposite edge is what makes tl/bl/tr resize feel natural (you offset left/top by the delta between old and new dimensions).
- **CSS custom property as a resize bridge:** When a widget's content area height is set via CSS (transition-driven expand/collapse), use a CSS custom property (`--bw-content-h`) as the bridge between JS resize logic and CSS layout. JS sets the property on the widget element during resize; CSS `var()` consumes it in both the `height` transition rule and the inner page height. This keeps the expand/collapse transition working normally while allowing resize to override the default.
- **CSS 3D flip ‚Äî instant reset on navigation:** When navigating between flashcards, if currently flipped, set `transition: 'none'` on the card inner, remove the flipped class, force a reflow with `void el.offsetHeight`, then restore transition to `''`. This prevents the user from seeing a backwards flip animation when moving between cards ‚Äî the reset is invisible and the new card appears front-facing.
- **CSS class-toggle transitions for content rotation:** Instead of Web Animations API, simple CSS transitions on opacity + transform driven by adding/removing classes (`ds-exit`, `ds-enter`) give clean two-phase swap animations. The trick: add exit class ‚Üí wait for transition ‚Üí swap content + add enter class ‚Üí force reflow ‚Üí remove enter class (letting the default non-transformed state be the transition target). Lighter than WAAPI for simple fade-slide swaps.
- **Progress bar reset trick:** Set `transition: none` + `width: 0`, force reflow with `void el.offsetWidth`, then set `transition: width Xms linear` + `width: 100%`. The reflow between no-transition and transition is what makes the reset invisible and the fill smooth.
- **CSS transitions kill live resize dragging.** If a child element has `transition: height 400ms` and you're updating its height target (via CSS custom property) on every mousemove, the 400ms transition means the element is perpetually lagging behind the mouse ‚Äî making it look like the resize is broken. Fix: set `element.style.transition = 'none'` on mousedown, restore to `''` on mouseup. This pattern applies to any drag interaction that fights a CSS transition.
- **Neon glow recipe:** Layered `text-shadow` is the entire trick ‚Äî 3-4 stops from tight (6-8px) to wide (50-80px), each progressively lower opacity. The color should match the text `color` exactly for realism. `box-shadow` with the same approach works for glowing borders/lines.
- **Neon flicker animation:** Irregular opacity dips at the tail end of a long cycle (4-5s) with very short dip durations (1-2% of cycle each). Multiple dips with varying depths (0.65-0.88) reads as electrical instability without being annoying. A separate gentler "hum" animation (0.97 opacity at 50%) on secondary elements adds ambient life.
- **Sign-switching blackout:** When rotating content in a neon sign, a 100-120ms full-opacity-0 gap between exit and enter animations sells the "sign powering off then back on" metaphor. Without it, the crossfade reads as a normal card transition.
- **macOS folder shape in pure CSS:** `::before` pseudo-element with `border-radius: 8px 8px 0 0` + `border-bottom: none` positioned at `top: -18px` creates a seamless folder tab. The 1px height overlap between tab and body hides the seam. Subtle blue-tinted border on both tab and body creates a cohesive folder "sheen" without needing actual blue fill. The tab being slightly lighter than the body (`#12121e` vs `#0f0f1a`) adds just enough depth to read as a separate plane.
- **Dynamic DOM refs for rebuildable UIs:** When a widget's innerHTML gets cleared and rebuilt (e.g., win state ‚Üí play again), `const` element refs go stale. Use getter functions (`function $el() { return document.getElementById('id'); }`) instead ‚Äî they always return the live element. Negligible perf cost for widgets with infrequent lookups.
- **CDN React + Babel standalone for no-build dashboards:** `<script type="text/babel">` with Babel standalone compiles JSX in-browser ‚Äî zero build step. Style objects over CSS-in-JS libs keeps the single-file constraint clean. CSS custom properties for the palette + inline style objects for component styles = best of both worlds. For developer tools, density > breathing room ‚Äî tight padding (6-8px rows), 11px font, monospace everywhere, and warm near-black backgrounds keep it functional without looking cold/clinical.
- **iOS glass material system:** The key to Apple-like dark UI is *layered translucency*, not flat surfaces. Base palette: near-void (#080503) ‚Üí glass layers at 4%/7%/11% warm-white opacity ‚Üí border at 8%/14%. The `inset box-shadow: 0 1px 0 rgba(255,244,234,0.06)` top highlight is what makes glass cards read as raised ‚Äî without it they look like flat rectangles. `backdrop-filter: blur(20px)` on cards, `blur(24px)` on sticky nav. Text hierarchy: 3 tiers of warm-cream at full/65%/35% opacity.

Initial data loaded via fetch *before* SSE connects to avoid blank state.

Entry animation: `scale(0.93) translateY(10px)` ‚Üí `scale(1)` with spring easing. Lock body scroll on mount (`overflow: hidden`), restore on unmount. Click-outside-to-close via `e.target === e.currentTarget` on backdrop. Escape key listener.

A 2px bar at the bottom animates `width: 100% ‚Üí 0%` over 3s linear ‚Äî gives the user a visual countdown without text. Multiple toasts offset via `60 + index * 50px`.
