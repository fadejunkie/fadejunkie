<objective>
## Goal
Add two discovery queries to the Convex backend and task Lobe with building a status discovery feed where users browse active statuses from other people in the ecosystem, filterable by path.

## Purpose
The status toggle system only becomes an ecosystem when users can *find each other* through their statuses. Without discovery, every toggle is a private signal that nobody sees. This plan is the bridge from "I broadcast my status" to "I find people whose status matches what I need" -- the core value proposition of the entire status system.

## Output
- `discoverStatuses` query added to `app/convex/statuses.ts` -- paginated, excludes current user, filterable by path and toggleKey, enriched with barber display info
- `getDiscoveryPaths` query added to `app/convex/statuses.ts` -- returns per-path count of active statuses for filter tabs
- Lobe task brief at `lobe/inbox/status-discovery-feed.md` for the discovery feed UI
</objective>

  <name>Task 1: Add discoverStatuses and getDiscoveryPaths queries to statuses.ts</name>
  <files>app/convex/statuses.ts</files>
  <action>
    Add two new exported queries to `app/convex/statuses.ts`, placed after the existing `getMyArchivedStatuses` query and before the `// -- Internal (cron-only) --` section.

    **Query 1: `discoverStatuses`**

    Args:
    - `path` (v.string()) -- required, which path to browse
    - `toggleKey` (v.optional(v.string())) -- optional filter to a specific toggle

    Implementation:
    - Get the authenticated userId via `getAuthUserId`. If not authenticated, return an empty array (don't throw -- discovery should gracefully degrade).
    - Query `statuses` using `by_path_active` index with `q.eq("path", path).eq("isActive", true)`, then `.collect()`.
    - Filter in JS: exclude statuses where `userId === currentUserId` (don't show your own), exclude where `expiresAt <= Date.now()` (expired but not yet archived by cron), and if `toggleKey` arg is provided, filter to only that toggleKey.
    - Sort by `activatedAt` descending (newest first).
    - Enrich each status with user display info: for each unique userId in the results, batch-lookup the barbers table via `by_userId` index. Map each status to include `{ barberName: string | null, barberSlug: string | null, barberAvatarUrl: string | null }`. Use a Map to avoid redundant lookups when multiple statuses share the same userId.
    - Also enrich with toggle config: `defaultDays` and `maxDays` from `getToggleConfig` (same pattern as `getMyStatuses`).
    - Return the enriched array. No pagination for now -- the `by_path_active` index scopes results per path, keeping result sets manageable at this stage.

    **Query 2: `getDiscoveryPaths`**

    Args: none

    Implementation:
    - Import `USER_PATHS` and the `PATH_LABELS` constant (you will need to define `PATH_LABELS` -- see below).
    - At the top of `statuses.ts` (below the existing imports), add a `PATH_LABELS` map:
      ```
      const PATH_LABELS: Record<string, string> = {
        barber: "Barbers",
        student: "Students",
        shop: "Shops",
        school: "Schools",
        vendor: "Vendors",
        event_coordinator: "Event Coordinators",
        client: "Clients",
      };
      ```
    - For each path in `USER_PATHS`, query `statuses` using `by_path_active` index with `q.eq("path", path).eq("isActive", true)`, collect, then filter where `expiresAt > Date.now()`.
    - Build an array of `{ path, label, count }` for each path.
    - Filter out paths with count === 0.
    - Sort by count descending.
    - Return the array.

    Do NOT modify any existing queries or mutations. Do NOT change the schema.
