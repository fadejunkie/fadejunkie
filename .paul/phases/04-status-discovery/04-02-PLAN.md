<objective>
## Goal
Build a complementary matching engine — a config of natural status pairs and a query that surfaces users whose active statuses complement the current user's active statuses. Task Lobe with a "Matches" UI section on the discover page.

## Purpose
Discovery (04-01) lets users browse statuses by path. But the real ecosystem payoff is *directed* discovery: a student with `seeking_apprenticeship` active doesn't need to manually browse shops — the system should surface shops that have `seeking_apprentice` active and hand-deliver the match. This turns passive browsing into active matchmaking, which is the entire thesis of the status toggle system.

## Output
- `COMPLEMENTARY_PAIRS` config array added to `app/convex/statusConfig.ts` — defines which status toggles are natural complements across (or within) paths
- `getComplementaryMatches` query added to `app/convex/statuses.ts` — takes the current user's active statuses, finds other users with complementary active statuses, returns enriched results
- Lobe task brief at `lobe/inbox/status-matches-ui.md` for a "Matches" tab/section on the discover page
</objective>

    Also add a helper function below the constant:
    ```ts
    /** Given a path + toggleKey, return all complementary targets (works both directions) */
    export function getComplementsFor(
      path: UserPath,
      toggleKey: string
    ): { path: UserPath; toggleKey: string }[] {
      const results: { path: UserPath; toggleKey: string }[] = [];
      for (const pair of COMPLEMENTARY_PAIRS) {
        if (pair.a.path === path && pair.a.toggleKey === toggleKey) {
          results.push(pair.b);
        } else if (pair.b.path === path && pair.b.toggleKey === toggleKey) {
          results.push(pair.a);
        }
      }
      return results;
    }
    ```

    Add a new exported query `getComplementaryMatches` to `app/convex/statuses.ts`, placed in the `// ── Discovery ──` section after `getDiscoveryPaths`.

    **Import addition:** Add `getComplementsFor` to the existing import from `./statusConfig`:
    ```ts
    import { getToggleConfig, USER_PATHS, getComplementsFor } from "./statusConfig";
    ```

    **Query: `getComplementaryMatches`**

    Args: none (uses authenticated user's own active statuses as input)

    Implementation:
    1. Get the authenticated userId via `getAuthUserId`. If not authenticated, return empty array.
    2. Get the caller's active statuses: query `statuses` with `by_userId` index, filter to `isActive === true` and `expiresAt > Date.now()`.
    3. For each of the caller's active statuses, call `getComplementsFor(status.path, status.toggleKey)` to get the list of complementary targets.
    4. For each complementary target `{ path, toggleKey }`, query `statuses` with `by_path_active` index (`path` + `isActive: true`), then filter in JS to match the specific `toggleKey`, exclude the current user, and exclude expired statuses.
    5. Deduplicate: use a `Set<string>` keyed by `statusId` to ensure a single status document is not returned multiple times (even if it matches multiple of the caller's statuses). When a match is first seen, record it. If seen again via a different caller status, skip it.
    6. Batch-lookup barber profiles: same pattern as `discoverStatuses` — collect unique userIds from all matches, query `barbers` table via `by_userId` index, build a `barberMap`.
    7. Return an array of match groups, one per active caller status that has matches:
       ```ts
       {
         myStatus: { path: string; toggleKey: string; _id: Id<"statuses"> };
         matches: Array<{
           _id: Id<"statuses">;
           userId: Id<"users">;
           path: string;
           toggleKey: string;
           activatedAt: number;
           expiresAt: number;
           barberName: string | null;
           barberSlug: string | null;
           barberAvatarUrl: string | null;
         }>;
       }
       ```
    8. Filter out groups with zero matches. Sort matches within each group by `activatedAt` descending.

    **Performance note:** This query fans out per-caller-status x per-complement-target. At current scale (early platform) this is fine. If it becomes expensive, the optimization path is a dedicated `by_path_toggleKey_active` compound index — but do NOT add that index now (no schema changes).

    Do NOT modify any existing queries or mutations. Do NOT change the schema.
  </action>
