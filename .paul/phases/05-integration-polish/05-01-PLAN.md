<objective>
## Goal
Add a public status summary query to the Convex backend and task Lobe with integrating active status badges across the three public-facing surfaces: barber profile, shop website, and community feed.

## Purpose
The status system is currently self-contained -- users activate statuses, browse discovery, send connections -- but none of that visibility reaches the surfaces people actually visit first. A barber's public profile shows no indication they are hiring, a shop website says nothing about booth availability, and feed posts give no signal that the author is actively seeking something. This plan bridges internal status state to external visibility. When someone lands on a barber profile or scrolls the feed, active statuses should be immediately visible as contextual signals -- turning every existing page into a passive discovery surface.

## Output
- `getPublicStatusSummary` query added to `app/convex/statuses.ts` -- takes a userId, returns active toggle labels grouped by path for public display
- Lobe task brief at `lobe/inbox/status-integration-badges.md` covering all three UI integration points (barber profile, shop website, feed posts)
</objective>

  <name>Task 1: Add getPublicStatusSummary query to statuses.ts</name>
  <files>app/convex/statuses.ts</files>
  <action>
    Add a new exported query `getPublicStatusSummary` to `app/convex/statuses.ts`. Place it after the existing `getActiveStatusesForUser` query and before the `// -- Discovery --` section.

    **Query: `getPublicStatusSummary`**

    Args:
    - `userId`: `v.id("users")`

    Implementation:
    1. This is a public query -- do NOT require authentication. Any visitor (logged in or anonymous) can see a user's active statuses on their public profile.
    2. Query `statuses` using `by_userId` index for the given userId. Collect all.
    3. Filter in JS: only statuses where `isActive === true` AND `expiresAt > Date.now()`.
    4. Group the filtered statuses by `path` field. For each path group, build an object:
       ```ts
       {
         path: string,           // e.g., "barber"
         pathLabel: string,      // e.g., "Barbers" — from PATH_LABELS constant
         statuses: Array<{
           toggleKey: string,    // e.g., "now_accepting_clients"
           label: string,        // humanized: "now accepting clients"
           expiresAt: number,
         }>
       }
       ```
    5. To humanize a toggleKey: replace all underscores with spaces. Example: `"seeking_booth_rental"` becomes `"seeking booth rental"`. This is a simple `toggleKey.replace(/_/g, " ")` -- no need for title case, keep it lowercase.
    6. Exclude path groups with zero active statuses (empty `statuses` array).
    7. Sort path groups by the count of active statuses descending (most active path first).
    8. Within each path group, sort statuses by `expiresAt` ascending (soonest to expire first -- creates urgency).
    9. Return the array.

    The `PATH_LABELS` constant already exists at the top of `statuses.ts` (added in plan 04-01). Reuse it.

    Do NOT modify any existing queries or mutations. Do NOT change the schema.
  </action>

    Brief content:

    **What:** Integrate active status badges across three existing surfaces -- barber profiles, shop websites, and community feed posts. Create a shared `PublicStatusBadges` component and wire it into each page. This makes the status ecosystem visible everywhere users already look.

    **New component:** `app/components/PublicStatusBadges.tsx`

    A reusable component that renders a user's active statuses as small, inline badges. This component will be used on all three surfaces.

    Props:
    - `statuses`: the return type of `api.statuses.getPublicStatusSummary` -- an array of `{ path, pathLabel, statuses: { toggleKey, label, expiresAt }[] }`
    - `compact` (optional boolean, default false): when true, show only the first 3 status labels as a condensed row (for feed posts where space is tight)

    Rendering:
    - Default mode: for each path group, render the pathLabel as a tiny uppercase label (`font-mono`, `text-[10px]`, `text-muted-foreground/60`, `tracking-widest`, `uppercase`), followed by the status labels as inline badges. Each badge: `font-mono`, `text-[11px]`, `px-2 py-0.5`, `rounded-full`, `bg-foreground/5`, `text-foreground/70`, `border border-border`. Keep it minimal -- these are signals, not CTAs.
    - Compact mode: flatten all statuses from all paths into a single row, take the first 3, render as small dot-separated labels. If more than 3, append "+N more" in muted text. No path group headers.
    - If the statuses array is empty or undefined, render nothing (return null).

    **Integration 1: Barber Profile (`app/barber/[slug]/page.tsx`)**

    This is a server-rendered page using `fetchQuery`. After the existing `fetchQuery` for the barber, add a second `fetchQuery` call:
    ```ts
    const statusSummary = await fetchQuery(api.statuses.getPublicStatusSummary, { userId: barber.userId });
    ```

    Render `<PublicStatusBadges statuses={statusSummary} />` in the hero section, after the barber's name/slug/location block and before the bio. Place it inside a `<div className="mt-3">` wrapper for spacing.

    **Integration 2: Shop Website (`app/shop/[userId]/page.tsx`)**

    This page has `userId` from the route params. Add a `fetchQuery` call:
    ```ts
    const statusSummary = await fetchQuery(api.statuses.getPublicStatusSummary, { userId: userId as Id<"users"> });
    ```

    Pass `statusSummary` as a new prop to `ShopTemplate`:
    ```tsx
    <ShopTemplate shop={...} statusSummary={statusSummary} />
    ```

    In `ShopTemplate.tsx`, accept the optional `statusSummary` prop and render `<PublicStatusBadges statuses={statusSummary} />` after the shop tagline and before the about section.

    **Integration 3: Community Feed Posts**

    The feed is client-rendered with paginated data from `listFeedPosts`. The enrichment approach here is different -- we cannot call a separate query per post (N+1 problem). Instead:

    Option A (recommended, simpler): Add a client-side component `PostStatusIndicator` that takes an `authorId` and uses `useQuery(api.statuses.getPublicStatusSummary, { userId: authorId })` to fetch that author's active statuses. Render as compact mode (`<PublicStatusBadges statuses={data} compact />`). Place it below the author name in the post card header. Convex's reactive queries will deduplicate across posts by the same author.

    Option B (stretch, only if A causes visible latency): Batch-fetch in the `listFeedPosts` query itself -- but this requires backend changes and is out of scope for this plan. Go with Option A.

    The `PostStatusIndicator` component:
    - Takes `authorId: Id<"users">`
    - Uses `useQuery(api.statuses.getPublicStatusSummary, { userId: authorId })`
    - If the query returns empty or undefined, renders nothing
    - If statuses exist, renders `<PublicStatusBadges statuses={data} compact />`
    - Must be a client component (`"use client"`)

    Place `PostStatusIndicator` inline in the feed post card, below the author name and above the post content.

    **Design constraints:**
    - Follow existing page patterns exactly -- barber profile uses `fetchQuery` server-side, feed uses `useQuery` client-side
    - Badges should feel like metadata, not decoration. Muted, small, monospaced. They enhance context without competing with primary content.
    - On the barber profile, status badges should feel like a natural extension of the hero section -- like services badges but more transient
    - On the shop website, respect the ShopTemplate's existing layout. The badges should sit naturally without disrupting the shop's brand feel.
    - On feed posts, compact mode keeps things tight. Status badges are secondary to the post content.
    - Mobile-first: badges should wrap naturally on narrow screens
    - No new dependencies
    - No new routes

    **Technical notes:**
    - `PublicStatusBadges` is a pure presentational component -- no data fetching. It receives pre-fetched data.
    - `PostStatusIndicator` is a thin client wrapper that fetches + renders `PublicStatusBadges` in compact mode
    - The `getPublicStatusSummary` query is public (no auth required), safe to call from server-rendered pages via `fetchQuery`
    - Import path for the query: `api.statuses.getPublicStatusSummary`
    - Import `Id` from `@/convex/_generated/dataModel` where needed for type casting
